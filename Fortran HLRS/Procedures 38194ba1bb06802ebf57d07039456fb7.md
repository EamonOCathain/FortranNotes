# Procedures

**See [functions](Functions%2038194ba1bb0680d69729ec69d7e35a77.md) and [subroutines](Subroutines%2038194ba1bb0680bf85edc8bae2a3166d.md) section for more**

# **Purpose of Procedures**

A **procedure** is a reusable block of code that performs a specific task.

In Fortran, procedures include:

- Functions
- Subroutines

Procedures are one of the most important tools for organizing scientific programs.

---

# **Why Use Procedures?**

As programs become larger, placing all code inside a single program quickly becomes difficult to manage.

Procedures allow a large program to be divided into smaller, meaningful parts.

Instead of writing:

```fortran
program huge_program

    ! thousands of lines of code

end program huge_program
```

we can organize code into separate procedures that each perform a specific task.

For example:

```fortran
call read_data(...)
call process_data(...)
call write_output(...)
```

This makes programs easier to understand and maintain.

---

# **Separation of Scope**

Each procedure has its own scope.

Variables declared inside a procedure only exist inside that procedure.

```fortran
subroutine example()

    integer :: i

end subroutine example
```

The variable `i` is not visible outside the subroutine.

Similarly, variables declared in a program are not automatically available inside a procedure unless:

- they are passed as arguments, or
- they are imported from a module.

This separation helps prevent accidental modification of variables.

---

# **SAVE Attribute**

Variables declared inside a procedure normally exist only while that procedure is executing.

The `save` attribute causes a variable to retain its value between procedure calls.

Example:

```fortran
subroutine counter()

    integer, save :: n = 0

    n = n + 1

    print *, n

end subroutine counter
```

Calling the procedure multiple times:

```fortran
call counter()
call counter()
call counter()
```

produces:

```
1
2
3
```

because `n` remembers its previous value.

Variables initialized during declaration automatically receive the `save` attribute:

```fortran
integer :: n = 0
```

is equivalent to

```fortran
integer, save :: n = 0
```

The `save` attribute is useful for storing persistent state, but excessive use can make procedures harder to understand and may reduce opportunities for parallel execution.

# **Dummy and Actual Arguments**

When a procedure is defined, its argument names are called **dummy arguments**.

```fortran
subroutine rectangle_area(length, width, area)
```

When the procedure is called, the supplied values are called **actual arguments**.

```fortran
call rectangle_area(x, y, A)
```

The names do not need to match.

```
length ↔ x
width  ↔ y
area   ↔ A
```

Arguments are matched by position.

---

# **What Can Be Passed to a Procedure?**

Procedure arguments can be many different types of objects.

Examples include:

- Variables
- Arrays
- Derived types
- Procedures (functions or subroutines)

Variable:

```fortran
call print_value(x)
```

Array:

```fortran
call compute_mean(values)
```

Derived type:

```fortran
call update_person(person)
```

Procedure:

```fortran
call integrate(f)
```

where `f` is itself a function.

---

# **Procedures Can Be Called Multiple Times**

A procedure can be called from many different places in a program.

```fortran
call process_data(data1)

call process_data(data2)

call process_data(data3)
```

The code only needs to be written once.

This reduces duplication and makes maintenance easier.

---

# **Reusability**

Procedures can be reused in different programs.

For example, a function stored in a module:

```fortran
module maths

contains

    function square(x) result(y)

        real, intent(in) :: x
        real :: y

        y = x**2

    end function square

end module maths
```

can be used by any program:

```fortran
use maths

a = square(4.0)
```

Reusable procedures are often collected into libraries.

---

# **Independent Testing**

Procedures can be tested independently from the rest of the program.

Example:

```fortran
function square(x) result(y)

    real, intent(in) :: x
    real :: y

    y = x**2

end function square
```

A small test program can verify that it works correctly:

```fortran
print *, square(5.0)
```

Output:

```
25.0
```

Testing procedures individually makes debugging much easier.

---

# **Reduced Complexity**

Breaking a program into procedures reduces complexity.

Instead of reading hundreds or thousands of lines of code, we can understand a program at a higher level:

```fortran
call read_data()
call process_data()
call write_results()
```

Each procedure can then be studied separately.

This approach:

- improves readability
- simplifies debugging
- encourages code reuse
- makes collaboration easier

---

# **Procedures in Derived Types**

Modern Fortran allows procedures to be associated with derived types.

Conceptually:

```fortran
type :: Person

contains

    procedure :: introduce

end type Person
```

This allows procedures to act on specific types of objects.

This topic belongs to object-oriented programming and is usually introduced later.

---

# **Example**

Without procedures:

```fortran
program example

    implicit none

    real :: x

    x = 5.0

    print *, x**2

end program example
```

With a procedure:

```fortran
program example

    implicit none

    real :: x

    x = 5.0

    print *, square(x)

contains

    function square(n) result(y)

        real, intent(in) :: n
        real :: y

        y = n**2

    end function square

end program example
```

The second version is easier to reuse and maintain.

---

# **Avoid Hidden Side Effects**

Modern Fortran assumes that procedures do not contain hidden side effects that interfere with the rest of the program.

A **side effect** is any change made by a procedure other than producing its intended result.

Examples of side effects include:

- Modifying global variables
- Changing variables passed to another procedure
- Altering module variables unexpectedly
- Depending on a particular order of evaluation

For example:

```fortran
function increment(x) result(y)

    real :: x
    real :: y

    counter = counter + 1   ! side effect

    y = x + 1

end function increment
```

The function not only computes a result but also changes the global variable `counter`.

Such behaviour can make programs difficult to understand and may prevent compiler optimizations.

In general, procedures should behave like mathematical functions:

```fortran
y = f(x)
```

The result should depend only on the inputs, and the procedure should not unexpectedly modify other parts of the program.

This is particularly important in scientific computing, where compilers aggressively optimize and parallelize code.

---

# **Procedure Independence and Optimization**

Fortran compilers assume that procedures are independent unless explicitly told otherwise.

This allows the compiler to:

- Reorder calculations
- Vectorize loops
- Execute operations in parallel
- Apply aggressive optimizations

For example:

```fortran
a = fun1(x) * fun2(y)
```

The compiler may evaluate:

```fortran
fun1(x)
```

first or

```fortran
fun2(y)
```

first.

The language does not specify an evaluation order.

Therefore:

```fortran
fun1()
```

must not modify data required by

```fortran
fun2()
```

and vice versa.

Programs that rely on a particular evaluation order may produce different results when compiled with different optimization settings.

# **Summary**

Procedures:

- Divide large programs into smaller parts.
- Create separate scopes for variables.
- Receive information through arguments.
- Can be called from multiple places.
- Can be reused in other programs.
- Can be tested independently.
- Reduce complexity and simplify debugging.
- May be associated with derived types.

---

# **Key Idea**

Procedures are the primary tool for organizing Fortran programs.

By separating a program into small, reusable functions and subroutines, code becomes easier to understand, test, maintain, and reuse.
